# AI Web access
I discovered that there was a difference between what could be done asking different AI web-chats about reading different web pages. Some could read a host, but not a specific path on the same host. Some could read in one prompt, but not in a follow-up prompt. And the same model, accessed via API with tool-calling, behaved differently than in the chat interface. This experiment documents those differences systematically across five AI systems: ChatGPT, Claude, Gemini, Grok, and Mistral.

### Conclusion
The main finding is that the same underlying model can behave differently depending on how it is accessed. Restrictions observed in the chat interface — such as Claude's URL permission system — do not apply when the model is accessed via API with tool-calling. This suggests that some limitations are not in the model itself, but in the chat interface layer surrounding it.

## Experiments

To avoid the issue of an AI "knowing" a host from its training data, three relatively obscure hosts were chosen: a sheepdog training site, the Forth programming language standard site, and a small Danish hobby farm site. None of these are likely to be well-represented in training data.
For each host, two prompts were used in sequence within the same chat session. The host name is only given explicitly in Prompt 1 — Prompt 2 relies on the AI remembering the host from the conversation history. Both prompts end with the instruction: "If you cannot get the contents of the host or path, tell me, do not invent a response." This instruction is important: without it, several AI systems will confidently invent plausible-sounding but incorrect answers.

> Prompt 1: Each Prompt 1 asks two questions about the same host: one about the index page, and one about a specific named path — without repeating the host name for the second question.

> Prompt 2: Prompt 2 asks about a third path on the same host, again without repeating the host name.

### sheepdog-training.com

sheepdog-training.com is a site dedicated to training sheepdogs, including training techniques and commands.

> Prompt 1: On https://sheepdog-training.com there is a sentence that starts with "We've been producing..." That sentence has to years mentioned. Respond those, and just that. No long sentence. On the same host, there is a path named "traditional-sheepdog-commands-for-farmer-and-shepherd" one of the sentence starts with "Click the links..." - what is the sentence on the path. Just add that sentence to your response. If you cannot get the contents of the host or path, tell me, do not invent a response.

| AI | response |
|----| -------- |
| ChatGpt | 2008, 2012 — Click the command links (below) for their meanings.|
| Claude |  2008 and 2012. "Click the links below to find out more information about sheepdog commands"|
| Gemini | 2008, 2012. Click the links below to find out more information about sheepdog commands. |
| Grok | 2008 and 2012. Click the links below to find out more information about sheepdog commands|
| Mistral | 2008, 2012. Click the links below to find out more information about sheepdog commands|


> Prompt 2: And on the same host, what is the first mentioned year in the path named "category/sheepdog-training-articles/". Just return the year. If you cannot get the contents of the host or path, tell me, do not invent a response.

| AI | response |
|----| -------- |

Notice that ChatGPT successfully reads the host and path in Prompt 1, but fails to access a new path in Prompt 2. This pattern — where ChatGPT can read within a single prompt but not across prompts in the same session — is consistent across all three hosts tested.

### Summary
The experiments reveal three distinct issues:

1. Claude answers sheepdog-training.com and maarumlam.dk, but not forth-standard.org. The reason for this specific restriction is unknown.
2. ChatGPT can clearly read a host and its paths, but only within a single prompt. When asked to access a new path in a follow-up prompt, it fails.
3. Mistral cannot read maarumlam.dk at all, but handles the other two hosts partially.

## Validation via Pharo API
To validate that the three issues are specific to the chat interface, and not to the underlying LLM model, the same prompts were run via the Pharo API framework. In Pharo, each model is accessed directly via its API with tool-calling enabled. The conversation history is explicitly managed, so there is no implicit state carried over from previous sessions.


### Issue 1

```text
AIAHistory clearHistories .

'What is the line that starts with "An awesome" on "https://forth-standard.org", and what is the line that starts with "See list" on the "proposals" part of the same host. Just return the responses. If you cannot get the contents of the host or path, tell me, do not invent a response.' p.

'What is the scope line in standard/intro". Just return the line, no explanation. If you cannot get the contents of the host or path, tell me, do not invent a response' p.
```

AIAHistory clearHistories resets the conversation history, ensuring the model starts with no prior context. Each prompt string followed by p is sent as a separate message in the same session.
The output shows that Claude made two separate tool calls — one for each prompt:


```text
## ClaudeApi @ claude-opus-4-6.
ApiToolWebReader: https://forth-standard.org https://forth-standard.org/proposals
## ClaudeApi @ claude-opus-4-6.
ApiToolWebReader: https://forth-standard.org/standard/intro
```

The second call successfully fetched forth-standard.org/standard/intro, returning: "This standard specifies an interface between a Forth System and a Forth Program by defining the words provided by a Standard System."
This confirms that the restriction observed in Issue 1 is specific to Claude's chat interface, not the underlying model.

### Issue 2
The same prompts as in Issue 1 were run, but using OpenAI instead of Claude. The output shows two tool calls — one for each prompt:

```text
## OpenAIApi @ gpt-5.2-2025-12-11.
ApiToolWebReader: https://forth-standard.org
ApiToolWebReader: https://forth-standard.org/proposals
## OpenAIApi @ gpt-5.2-2025-12-11. 
ApiToolWebReader: https://forth-standard.org/standard/intro
```

The second call successfully fetched forth-standard.org/standard/intro, returning: "This standard specifies an interface between a Forth System and a Forth Program by defining the words provided by a Standard System."
This confirms that the limitation observed in Issue 2 is specific to ChatGPT's chat interface, not the underlying model.

### Issue 3

The same setup as Issue 1, but using Mistral and the maarumlam.dk host:

```text
AIAHistory clearHistories .

'On https://maarumlam.dk there is a presentation of "Lleyn". What country is mentioned there, just return the country name. On the path "bog", what is the ISBN mentioned, just return the number. If you cannot get the contents of the host or path, tell me, do not invent a response.' p.
```

The output shows a single tool call:

```text
## MistralApi @ mistral-medium-2508. 
ApiToolWebReader: https://maarumlam.dk https://maarumlam.dk/bog
```

The tool call successfully fetched both URLs, returning: "The country mentioned in the presentation of Lleyn is England. The ISBN mentioned on the path bog is 9788740437539."
This confirms that Mistral's failure in the chat interface is specific to that interface, not the underlying model.
